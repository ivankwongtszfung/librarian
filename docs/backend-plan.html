Librarian · Backend Plan v1 · 2026-07-12 · scope: Phase 1 (+ Phase 2 hooks)

Backend Plan — the daemon behind the library

One Node/TS process on the Mac: MCP gateway for agents, JSON API + SSE for the web and iOS apps, transcript watcher for auto-capture, SQLite as the permanent decision store. Planned BDD-first: every milestone ships when its scenarios pass.

01Scope & non-goals

In scope (Phase 1): store + schema · MCP tools (submit_for_review, get_review, record_decision, search_decisions, get_constraints) · REST + SSE for clients · transcript watcher (auto-capture) · verdict/comment endpoints (so the apps work end-to-end) · notification hook to ntfy.

Hooks for Phase 2, built but minimal: version lineage + diff endpoint, verdict state machine with reasons (red lights recorded from day one).

Non-goals now: reviewer panel (librarian agent), enrichment/ripgrep verification, Discord surface, tmux/native-prompt adapter, retro-mining the 750 MB backlog.

02Backend ADRs

Dogfooding our own thesis: rejected alternatives are recorded with the same weight — these red lights are load-bearing decisions.

ADR-B1Node/TS, single daemon processaccepted

One process serves MCP, REST/SSE, and the watcher. Official MCP TS SDK; one language end-to-end with the web client.

Rejected: Python/FastMCP (second stack for API+web), Go (slower iteration on an API-heavy product).

ADR-B2SQLite (WAL) + FTS5 as the system of recordaccepted

Zero-ops, single file on the user's disk, FTS5 covers search_decisions. WAL for concurrent readers while the watcher writes.

Rejected: Postgres (ops burden, nothing needed), graph DB as primary (kept as later export target for codebase-memory-mcp).

ADR-B3MCP transport: streamable HTTP, one shared serveraccepted

All agent sessions point at http://127.0.0.1:7801/mcp. Shared state trivially; a 20-line stdio→HTTP shim ships for stdio-only agents.

Rejected: stdio-per-session (N processes sharing one DB, and the apps still need a daemon anyway).

ADR-B4Verdict delivery: server-side long-pollaccepted

get_review(review_id, wait_seconds≤50) holds the request server-side, returns pending on timeout; agent re-calls. Few calls, no busy-loop, safe under every client's tool timeout. Tool description instructs agents to back off between polls.

Rejected: one blocking call (MCP clients time out inconsistently → agent guesses), naive fast polling (token burn), MCP elicitation (routes to the agent's own UI — wrong screen by definition).

ADR-B5Dual ingestion: MCP push + transcript watcheraccepted

Agents that follow the convention push via MCP; the watcher tails ~/.claude/projects/**/*.jsonl and auto-captures approved ExitPlanMode plans and docs/** writes so nothing is missed. Content-hash dedupe merges the two paths.

Rejected: MCP-only (forgetful agents create silent gaps in the library — the product promise is "save every decision").

ADR-B6Client API: REST + SSEaccepted

Plain REST for reads/verdicts/comments; one SSE stream for live updates. Trivially consumable from fetch and URLSession.

Rejected: WebSocket (no duplex need), GraphQL (schema ceremony for two known clients).

ADR-B7Enrichment v1 via ripgrep subprocessproposed · Phase 2

Reference verification (phantom-file/function flags) shells out to rg; codebase-memory-mcp becomes an optional backend later. Interface designed now, implemented later.

ADR-B8GitHub PRs as the review transportrejected

Rejected 2026-07-11: no cross-session triage, no agent-queryable red-light memory, requires network+GitHub for a local-first tool. Kept as a zero-code pilot idea only.

ADR-B9tmux scraping as the foundation layerrejected

Rejected 2026-07-11: brittle, Claude-only, and Omnara already ships native-prompt remote approval. Demoted to an optional future adapter; the foundation speaks MCP only.

Verdicts are an append-only event table, not a column update — the red-light history (including a rejection later revised into an approval) is never overwritten. Decision.status is a denormalized cache of the latest event.

-- core DDL sketch (full migrations in M1)
CREATE TABLE decisions (
  id TEXT PRIMARY KEY, project_id TEXT NOT NULL, session_id TEXT,
  kind TEXT CHECK (kind IN ('plan','adr','prd','arch')),
  title TEXT NOT NULL, status TEXT NOT NULL DEFAULT 'pending',
  source TEXT CHECK (source IN ('mcp','watcher')),
  content_hash TEXT UNIQUE, pinned_commit TEXT,
  created_at INTEGER NOT NULL, decided_at INTEGER
);
CREATE TABLE verdict_events (
  id TEXT PRIMARY KEY, decision_id TEXT NOT NULL REFERENCES decisions(id),
  from_state TEXT NOT NULL, to_state TEXT NOT NULL,
  participant_id TEXT NOT NULL, reason TEXT, at INTEGER NOT NULL,
  CHECK (to_state != 'rejected' OR reason IS NOT NULL)  -- red lights carry reasons
);
CREATE VIRTUAL TABLE decisions_fts USING fts5(title, body, reason);

06API surface

MCP tools (agents)

ToolBehavior
submit_for_reviewproject, title, doc, kind, parent_review_id?, context_refs? → review_id. Creates pending Decision+Version, fires notify + SSE.
get_reviewreview_id, wait_seconds≤50 → {status} | {verdict, reason, comments[]}. Long-poll per ADR-B4.
record_decisionFYI tier: stores as approved/informational, no gate, no notify.
search_decisionsquery, project?, filters (status·kind·date) → ranked FTS5 hits incl. rejections + reasons. The mid-design lookup: a specific question against the full archive.
get_constraintsproject, topic? → bounded digest of accepted + rejected (with reasons), newest first. The pre-design briefing: queryless on purpose — agents can't search for constraints they don't know exist. Kept separate from search so tool descriptions teach distinct call moments.

07BDD scenarios (acceptance = these pass)

Run with cucumber-js against an in-process daemon, temp SQLite, and fixture transcript files. Tags map to milestones below.

Feature: Submit for review                                   @M2
  Scenario: Agent submits a plan
    Given the daemon is running with an empty store
    When an agent calls submit_for_review with a plan for "accounting_app"
    Then a decision exists with status "pending" and source "mcp"
    And an SSE event "decision.added" is emitted
    And a notification is published to the ntfy topic

  Scenario: Revision links lineage                            @M4
    Given a decision in state "changes_requested" with version 1
    When the agent resubmits with parent_review_id
    Then version 2 exists with parent_version_id = version 1
    And GET /diff?from=1&to=2 returns a non-empty unified diff

Feature: Long-poll verdict delivery                           @M4
  Scenario: Poll returns pending on hold expiry
    Given a pending decision
    When the agent calls get_review with wait_seconds 2
    Then the call returns after ~2s with status "pending"

  Scenario: Verdict arrives during hold
    Given a pending decision and an in-flight get_review
    When a human posts verdict "rejected" with reason "conflicts with ADR-004"
    Then the in-flight call resolves with the verdict and the reason
    And the decision's verdict_events contain pending → rejected

  Scenario: Connection drops during hold — verdict is never lost
    Given a pending decision and an in-flight get_review
    When the connection drops before the hold expires
    And a human posts verdict "approved" while no poll is connected
    Then the verdict_event is persisted normally
    When the agent reconnects and calls get_review again
    Then the call returns immediately with the stored verdict
    And repeated get_review calls return the same result (idempotent read)

  Scenario: Daemon restarts while a review is pending
    Given a pending decision with review_id "R"
    When the daemon restarts
    And the agent calls get_review with "R"
    Then the call succeeds against the recovered store
    And the decision is still listed as "pending" in the library

Feature: Watcher auto-capture                                 @M5
  Scenario: Approved plan is captured without agent cooperation
    Given a fixture transcript receiving an approved ExitPlanMode entry
    Then within 2s a decision exists with kind "plan" and source "watcher"
    And its session external_ref points at the transcript

  Scenario: Dedupe across ingestion paths
    Given a decision submitted via MCP with content hash H
    When the watcher captures a doc with the same hash H
    Then exactly one decision exists, with both provenances recorded

Feature: Red-light memory                                     @M3
  Scenario: Rejections are first-class in search
    Given a decision rejected with reason "overkill for single user"
    When an agent calls search_decisions "microservices"
    Then the hit includes status "rejected" and the reason verbatim

  Scenario: Constraints digest
    When an agent calls get_constraints for "accounting_app"
    Then the digest lists accepted and rejected decisions with reasons

Feature: Verdict API guards                                   @M4
  Scenario: Reject without reason is refused
    When a client posts verdict "rejected" with no reason
    Then the API responds 422 and no verdict_event is written

09Risks & mitigations

  • Transcript format drift (watcher parses undocumented JSONL): keep extractors as small pure functions with fixture files per Claude Code version; a parse failure degrades to "unparsed event" — never crashes ingestion.
  • Agent ignores the convention: watcher is the safety net for capture; enforcement hook (deny ExitPlanMode without review) stays in the Phase 2 drawer.
  • Long-poll vs client timeouts / dropped connections: wait capped at 50s, configurable down; tool description advises backoff cadence. Connection loss is benign by construction — the poll is a stateless idempotent read; verdicts persist in SQLite regardless of who's connected, so the next poll (or a daemon restart) always recovers. Pinned by the @M4 reconnect scenarios.
  • SQLite write contention (watcher + API): WAL + a single serialized writer queue in-process.
  • Security: bind 127.0.0.1 + Tailscale interface only; bearer token for REST; reject-reason and every verdict logged append-only.
