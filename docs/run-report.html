02 · AcceptanceThe scenarios are the spec

The .feature files aren't decoration — they drive a real daemon over real HTTP with a real MCP client. No mocks in the middle. If a scenario passes here, an agent can do it.

The two reconnect scenarios exist to pin a property that a future refactor could quietly break: the long-poll is an idempotent read reconstructed from committed rows. Losing the connection costs latency, never a verdict.

03 · DogfoodThe first real decision

Librarian's first use case was its own deployment question. I wrote ADR-001: How Librarian is deployed — comparing manual startup, a launchd/systemd user service, Docker, a hosted service, and a hosted relay — then submitted it through Librarian's own MCP server, exactly as an agent would.

  1. Agent asks what it must know. get_constraints("librarian") → empty. Nothing decided yet.
  2. Agent submits the ADR. submit_for_review → dec_9cb19238…, status pending.
  3. Agent blocks. get_review → "Still awaiting the human." The session is now genuinely gated on a person reading the doc.
  4. Human reads and approves. The recommendation: a per-user background service, with Tailscale demoted from hard dependency to an optional tier. Docker and hosted-SaaS rejected — both destroy the trust layer, because a daemon that can't read your repos can't verify a design's claims.
  5. A second agent proposes Docker anyway — it wasn't in the conversation and has no memory of it.
  6. Red light, with a reason. A reasonless rejection is refused first (422). Then the real one lands.
  7. A third, fresh agent asks what it must know. And this is the whole product.
$ curl -X POST .../verdict -d '{"to":"rejected"}' HTTP 422 ← a red light with no reason is not a decision, it's a shrug $ curl -X POST .../verdict -d '{"to":"rejected","reason":"bind-mounting the home dir …"}' {"ok":true,"status":"rejected"} $ agent-3 → get_constraints("librarian") { "accepted": [ { "title": "ADR-001: How Librarian is deployed", "kind": "adr" } ], "rejected": [ { "title": "Ship Librarian as a Docker container", "reason": "bind-mounting the home dir to restore the trust layer defeats the point of the container (see ADR-001)" } ], "guidance": "The rejected items are standing red lights. Re-proposing one without addressing its reason will be rejected again." }
The thesis, working An agent that was never part of the conversation now knows what was turned down — and why — before it designs anything. The rejection didn't scroll away. It became a constraint.

04 · FoundTwo bugs the process caught

Both were invisible to the test suite as written. One was caught by insisting the process exit cleanly; the other only by running the built artifact instead of the source.

The daemon could not shut down

The long-poll and SSE need Node's 300s requestTimeout disabled — otherwise the server kills its own held requests. But that also means idle keep-alive sockets never expire, so server.close() waits forever for them. npm test hung after passing; a real daemon would have hung on SIGTERM. Fixed with an explicit closeAllConnections(), plus unref() on the in-flight hold timer.

The built package had no database

tsc doesn't copy .sql files, so dist/store/migrations/ didn't exist and the built daemon died on first boot: ENOENT: no such file or directory. Every test passed anyway, because vitest and cucumber run from source via tsx. Only the end-to-end smoke test against dist/ exposed it. Fixed with a build:assets step.

A third issue was a genuine design correction, caught by a failing scenario: a revision was minting a new decision instead of adding v2 to the existing one. The plan was explicit — one decision is one review thread — and the fix means the doc, the comments, and the red light that prompted the rewrite all stay together. The state machine now allows rejected → pending, which is exactly the rejection lineage the PRD celebrates: rejected → revised → approved, with none of it overwritten.
